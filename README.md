# Claude Code Airgap

Offline staging and deployment tooling for Claude Code.

Phase 1 target platforms:
- Windows x64
- Linux x64 with glibc

Out of scope in phase 1:
- ARM64
- musl/Alpine
- TUI

The tool is split into two operational phases:
- `stage`: run on an online machine that can reach the official Claude release bucket
- `deploy`: run on an offline or restricted machine using the prepared bundle

## What This Repository Does

This repository provides:
- official release download from the Claude Code release bucket
- official `manifest.json` checksum and size verification
- offline bundle preparation
- offline deployment from a copied bundle
- PATH setup for the installed `claude` command
- generated official `settings.json`
- conservative settings handling
- post-install health checks

This repository does not:
- implement a custom package format
- rehost Claude binaries
- bypass the vendor installer
- guarantee that your runtime model endpoint is reachable after install

## Repository Distribution Policy

This repository follows the Eldrun-style separation model:
- `claude-code-airgap`: offense360 private source repo
- `claude-code-airgap-public`: offense360 public distribution repo
- `claude-code-airgap-kangwonland`: Kangwonland public distribution repo

Public repos receive only the whitelist defined in `.publish-manifest`.

Included in public repos:
- stage and deploy scripts
- settings template
- operator-facing docs

Excluded from public repos:
- `downloads/`
- `docs/superpowers/specs/`
- `docs/superpowers/plans/`
- other internal working material not listed in `.publish-manifest`

The public distribution repos are intended to be content-identical.

## Support Matrix

### Windows

- Supported shell: PowerShell 5.1 or PowerShell 7+
- Supported artifact: `win32-x64`
- Git for Windows is still relevant for Claude Code runtime behavior on Windows

### Linux

- Supported shell: `bash`
- Required commands: `curl`, `sha256sum`, `ldd`, `grep`, `sed`, `awk`
- Supported artifact: `linux-x64`
- Only glibc-based systems are supported

### Rejected by design

- ARM64 Windows
- ARM64 Linux
- musl/uClibc systems such as Alpine

## Bundle Layout

The tool writes a deployable bundle into `downloads/`.

Example:

```text
downloads/
├── VERSION.json
├── manifest.json
├── claude-2.1.97-win32-x64.exe
└── claude-2.1.97-linux-x64
```

The deploy script can locate the bundle in either of these layouts:
- bundle files directly next to the deploy script
- bundle files inside a `downloads/` subdirectory next to the deploy script

## Metadata Files

### `manifest.json`

- downloaded from the official Claude Code release bucket
- used as the checksum and size source of truth
- rechecked during deploy

### `VERSION.json`

- generated by `stage`
- stores:
  - tool version
  - Claude version
  - download timestamp
  - staged platforms
  - source URLs

Important behavior:
- staging another platform for the same Claude version updates `downloaded_platforms`
- staging a different Claude version into the same `downloads/` directory is rejected
- if you want a different Claude version, clear `downloads/` first

## Operational Model

### Online machine

Use `stage` on a machine that can reach:
- `https://storage.googleapis.com`
- the official Claude Code release bucket

### Offline machine

Copy the repository folder or the bundle contents to the offline machine and run `deploy`.

The deploy scripts:
- validate bundle metadata
- validate binary size and SHA256 again
- prepare PATH
- create or modify the official `settings.json`
- execute the verified native installer
- run post-install health checks

### Stage Behavior

`stage` performs these steps:
1. Validate arguments
2. Resolve Claude version
3. Download `manifest.json`
4. Download requested platform binaries
5. Verify downloaded size
6. Verify SHA256 against `manifest.json`
7. Write or update `VERSION.json`

If a file already exists and matches the expected size and checksum:
- it is reused
- it is not downloaded again

If a file exists but does not match:
- it is deleted
- it is downloaded again

If a download fails integrity checks:
- the partial file is removed
- the command exits with an error

### Deploy Behavior

`deploy` performs these steps:
1. Detect the local platform
2. Locate `VERSION.json` and `manifest.json`
3. Confirm the current platform is included in the bundle
4. Confirm the staged Claude version matches the manifest version
5. Recompute SHA256 and size for the local platform binary
6. Copy the verified binary to a tool-controlled temporary working directory
7. Run the verified installer
8. Configure PATH
9. Configure `settings.json`
10. Run health checks

## PATH Handling

### Windows

Target PATH entry:

```text
%USERPROFILE%\.local\bin
```

Behavior:
- if the entry is missing from the User PATH, deploy adds it
- the current process PATH is also updated so the current shell can resolve `claude`

### Linux

Target PATH entry:

```text
$HOME/.local/bin
```

Behavior:
- deploy exports the path for the current process
- deploy appends a small managed PATH block to:
  - `~/.bashrc`
  - `~/.profile`

## `settings.json` Handling

The tool uses the official Claude Code user settings file:

- Windows: `%USERPROFILE%\.claude\settings.json`
- Linux: `$HOME/.claude/settings.json`

The tool does not create or use `claude.json`.

### Managed default keys

The generated template currently contains:

```json
{
  "env": {
    "DISABLE_AUTOUPDATER": "1",
    "CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC": "1",
    "DISABLE_NON_ESSENTIAL_MODEL_CALLS": "1",
    "ANTHROPIC_BASE_URL": "http://127.0.0.1:4000",
    "ANTHROPIC_AUTH_TOKEN": "no-token"
  }
}
```

Meaning:
- `DISABLE_AUTOUPDATER=1`
  - disables native background auto-update behavior
- `CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC=1`
  - disables nonessential traffic such as telemetry-style features
- `DISABLE_NON_ESSENTIAL_MODEL_CALLS=1`
  - reduces nonessential model-driven requests
- `ANTHROPIC_BASE_URL=http://127.0.0.1:4000`
  - points Claude Code to a local or proxied Anthropic-compatible gateway
- `ANTHROPIC_AUTH_TOKEN=no-token`
  - placeholder token value for gateways that do not require a real token

### Windows settings policy

Windows is conservative but allows limited merge.

If `%USERPROFILE%\.claude\settings.json` does not exist:
- deploy creates it from the template

If the file already exists:
- deploy parses it as standard JSON
- deploy fails if the file is invalid JSON
- deploy fails if `env` exists but is not an object
- deploy merges only managed keys under `env`
- unrelated existing keys remain
- deploy creates a timestamped backup before writing

### Linux settings policy

Linux is fail-closed by default.

If `$HOME/.claude/settings.json` does not exist:
- deploy creates it from the template

If the file already exists:
- deploy stops without modifying it
- to replace it intentionally, set:

```bash
export CLAUDE_CODE_AIRGAP_REPLACE_SETTINGS=1
```

- when replacement is allowed, deploy writes a timestamped backup first

## Gateway Template Behavior

The default gateway placeholders are intentionally simple.

If your local gateway uses a different address:
- change `ANTHROPIC_BASE_URL`

If your local gateway requires authentication:
- replace `ANTHROPIC_AUTH_TOKEN=no-token` with the real token

If your gateway does not require authentication:
- `no-token` can remain as a harmless placeholder if your gateway ignores it

## Health Checks

After installation, deploy runs:

- `claude --version`
  - required health check
  - if this fails, deploy fails

- `claude doctor`
  - best-effort health check
  - a non-zero result is shown as a warning
  - deploy does not fail only because `doctor` returned non-zero

This distinction exists because offline or gateway-based environments may cause `doctor` to warn even when the native installation itself is correct.

## Update Workflow

Auto-update is intentionally disabled.

Updates are manual:
1. On an online machine, run `stage` again with the new Claude version
2. Copy the updated bundle to the offline machine
3. Run `deploy` again on the offline machine

Behavior notes:
- if you stage a new platform for the same Claude version, the bundle metadata accumulates both platforms
- if you want to switch to a different Claude version, clear `downloads/` before staging

## Safety Notes

- Deploy revalidates checksum and size before installation.
- The tool does not assume the vendor's final install path.
- The verified binary is copied to a temporary tool-controlled working directory and executed from there.
- Linux supports glibc x64 only.
- Windows and Linux both reject ARM64 in phase 1.
- Existing invalid settings files are not auto-repaired.
- Linux will not silently overwrite an existing `settings.json`.

## Troubleshooting

### Stage says unsupported platform

Cause:
- you supplied an unsupported platform token
- or you are running on ARM64

Action:
- use only `win32-x64` and `linux-x64`

### Stage fails because version mismatch exists in `downloads/`

Cause:
- the existing `downloads/` directory contains a different Claude version

Action:
- remove `downloads/`
- rerun stage for the desired version

### Deploy cannot find bundle metadata

Cause:
- `VERSION.json` and `manifest.json` are not next to the deploy script
- or they are not under `downloads/`

Action:
- place bundle files in one of the supported layouts

### Windows deploy fails on existing settings

Cause:
- `%USERPROFILE%\.claude\settings.json` is invalid JSON
- or its `env` value is not an object

Action:
- inspect the file
- correct it manually
- rerun deploy

### Linux deploy refuses to overwrite settings

Cause:
- `$HOME/.claude/settings.json` already exists

Action:
- back it up manually and remove it
- or run with `CLAUDE_CODE_AIRGAP_REPLACE_SETTINGS=1`

### `claude doctor` warns after deploy

Cause:
- offline runtime
- gateway reachability problem
- authentication mismatch

Action:
- check your gateway endpoint
- check your token
- confirm `claude --version` still succeeds
